## Q1: 测试拓扑中的设备角色有哪些？

## Q2: 两台设备的具体角色分工是什么？

### My Recommendation
在双设备互测模式下，角色分工通常有两种方式：

**方式A：固定角色分工**
- DUT：始终作为接收端，被动接收流量，监控网卡性能指标
- Tester：始终作为发送端，主动发包，模拟各种流量场景
- 优点：职责清晰，便于统计DUT端指标
- 缺点：无法测试DUT作为发送端的场景

**方式B：双向互测**
- DUT和Tester轮流作为发送/接收端
- 每个测试用例可以指定流量方向（DUT→Tester 或 Tester→DUT）
- 优点：覆盖双向场景，更全面
- 缺点：测试配置更复杂

**推荐：方式A（固定角色分工）**，因为网卡稳定性测试的核心是验证被测网卡在各种流量压力下的表现，DUT作为接收端更符合"被动承受压力"的测试意图。

### User's Answer
**方式A：固定角色分工**
- DUT：始终作为接收端，被动接收流量，监控网卡性能指标
- Tester：始终作为发送端，主动发包，模拟各种流量场景

**决策理由**：职责清晰，便于统计DUT端指标，符合网卡稳定性测试的核心目标

## Q4: 网络连接方式是什么？

### My Recommendation
两台设备的网络连接方式直接影响测试场景的真实性和干扰因素：

| 连接方式 | 描述 | 优点 | 缺点 | 适用场景 |
|----------|------|------|------|----------|
| **A. 直连** | DUT网卡 ↔ Tester网卡，网线直连 | 无交换机干扰，纯网卡性能测试 | 无法测试交换机交互场景 | 网卡驱动/硬件稳定性验证 |
| **B. 经交换机** | DUT ↔ 交换机 ↔ Tester | 更接近真实网络环境 | 交换机成为干扰变量，影响测试结果 | 数据中心实际部署场景验证 |

**推荐做法**：
1. **基础稳定性测试（压力/长时间）**：使用直连模式，排除交换机干扰
2. **业务流测试**：可选经交换机模式，模拟真实部署场景
3. 配置文件支持连接方式参数，用例可选择连接模式

### User's Answer
**支持两种连接方式**：
- 直连模式（direct）：DUT网卡 ↔ Tester网卡，网线直接连接
- 经交换机模式（switch）：DUT ↔ 交换机 ↔ Tester

**设计要点**：
1. 配置参数：`connection_mode: "direct" | "switch"`
2. 压力测试/长时间测试：优先使用直连模式（排除干扰）
3. 业务流测试：可选经交换机模式（模拟真实场景）
4. 测试结果按连接方式分类统计

## Q5: 是否需要独立监控节点？

### My Recommendation
监控系统性能指标（CPU、内存、网卡流量）有两种方式：

| 方式 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 本地监控** | DUT本地运行监控脚本，收集自身指标 | 简单，无需额外设备 | 监控脚本消耗DUT资源，可能影响测试精度 |
| **B. 独立Monitor节点** | 第三台设备远程SSH连接DUT，采集指标 | 不消耗DUT资源，数据更准确 | 需要额外设备，增加部署复杂度 |

**推荐：独立Monitor节点**（如果设备资源允许）

原因：
1. 网卡稳定性测试需要精确的性能指标，本地监控脚本可能引入额外负载
2. 独立Monitor可以同时监控DUT和Tester两端指标
3. 长时间测试（7x24h）需要持续监控，独立节点更稳定

**如果设备资源有限，本地监控也是可行方案**。

- 使用Prometheus + node_exporter监控数据
- DUT和Tester都安装node_exporter
- Prometheus服务器采集指标数据

**设计要点**：
1. DUT和Tester安装node_exporter（标准端口9100）
2. Monitor节点部署Prometheus服务器
3. 测试期间Prometheus采集指标：
   - CPU使用率、内存使用率
   - 网卡流量（rx_bytes、tx_bytes）
   - 网卡丢包（rx_dropped、tx_dropped）
   - 网卡错误（rx_errors、tx_errors）
4. 测试框架通过Prometheus API查询指标数据
5. 测试报告包含Prometheus采集的性能指标图表

## Q7: 流量生成工具选择？

### My Recommendation
Linux环境下主流流量生成工具对比：

| 工具 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **iperf3** | 标准网络性能测试工具 | 功能全面（TCP/UDP/多线程）、参数丰富、社区成熟 | UDP模式不支持包大小自定义限制 | 通用性能测试、吞吐量测试 |
| **netperf** | 老牌网络性能工具 | 支持多种测试模式 | 已停止维护，不如iperf3流行 | 旧系统兼容场景 |
| **pktgen** | Linux内核模块，高速发包 | 极高性能，可达到网卡极限 | 配置复杂，需要内核模块操作 | 极高压压力测试、网卡极限验证 |
| **Moongen** | 基于MoonGen的高性能发包工具 | 10G+网卡高性能测试 | 需要特殊硬件支持（DPDK） | 高性能网卡专项测试 |
| **tcpreplay** | 回放pcap文件 | 可复现真实流量模式 | 需要预先录制流量包 | 业务流模拟测试 |

**推荐组合**：
1. **主力工具：iperf3** - 覆盖压力测试、长时间测试、吞吐量测试
2. **补充工具：pktgen** - 用于极限压力测试（iperf3无法达到网卡极限时）
3. **补充工具：tcpreplay** - 用于业务流测试（回放真实流量模式）

### User's Answer
**多驱动场景测试**：
- 内核驱动（Kernel Driver） → 使用 **iperf3**
- DPDK驱动 → 使用 **pktgen**
- RDMA驱动 → 使用 **perftest**

**设计要点**：
1. 测试框架需要支持多驱动场景
2. 根据驱动类型自动选择对应的流量生成工具
3. 配置参数：`driver_type: "kernel" | "dpdk" | "rdma"`
4. 工具映射：
   - kernel → iperf3（TCP/UDP流量）
   - dpdk → pktgen（DPDK高性能发包）
   - rdma → perftest（RDMA性能测试，ib_write_bw/ib_send_bw等）
5. 不同驱动类型的测试指标可能不同：
   - kernel：吞吐量、丢包率、延迟、CPU使用率
   - dpdk：吞吐量、包转发率、CPU使用率（DPDK独占CPU核）
   - rdma：吞吐量、延迟、带宽利用率、RDMA操作类型性能

## Q8: 驱动类型如何识别？

### My Recommendation
多驱动场景下，测试框架需要自动识别当前使用的驱动类型：

| 驱动类型 | 识别方式 | 命令示例 |
|----------|----------|----------|
| **内核驱动** | 检查网卡是否绑定到内核网络栈 | `ethtool -i eth0` 显示driver信息；网卡出现在`ifconfig`或`ip link`输出中 |
| **DPDK驱动** | 检查网卡是否绑定到DPDK（vfio/uio） | `dpdk-devbind.py --status`；网卡从内核网络栈消失 |
| **RDMA驱动** | 检查RDMA设备是否存在 | `ibv_devices`或`rxe-dev`命令；RDMA设备出现在`/sys/class/infiniband/` |

**推荐：配置文件指定 + 自动检测双重机制**

设计：
1. **配置文件优先**：`driver_type: "kernel"` 明确指定驱动类型
2. **自动检测兜底**：如果配置未指定，通过上述命令自动识别
3. **驱动切换测试场景**：如果需要测试驱动切换（如kernel→dpdk），需要提供驱动绑定脚本

### User's Answer
**仅配置文件指定驱动类型**：
- 不做自动检测
- 配置参数明确指定：`driver_type: "kernel" | "dpdk" | "rdma"`
- 测试框架直接读取配置，根据驱动类型选择工具

**设计要点**：
1. 配置文件必填参数：`driver_type`
2. 工具映射（硬编码）：
   - kernel → iperf3
   - dpdk → pktgen
   - rdma → perftest
3. 配置错误时（如指定dpdk但工具未安装）直接报错，不做自动修正
4. 优点：简单可靠，避免检测逻辑出错

## Q9: RDMA测试具体用哪些perftest命令？

### My Recommendation
perftest工具包含多种RDMA操作测试命令：

| 命令 | RDMA操作类型 | 描述 | 适用场景 |
|------|-------------|------|----------|
| **ib_write_bw** | RDMA Write | 单向写操作，测试写带宽 | 最高带宽场景（Write延迟最低） |
| **ib_read_bw** | RDMA Read | 单向读操作，测试读带宽 | 读密集场景 |
| **ib_send_bw** | RDMA Send/Receive | 双向发送接收，测试Send带宽 | 需要双向确认的场景 |
| **ib_write_lat** | RDMA Write | 测试写延迟（小消息） | 延迟敏感场景 |
| **ib_read_lat** | RDMA Read | 测试读延迟 | 延迟敏感场景 |
| **ib_send_lat** | RDMA Send | 测试Send延迟 | 延迟敏感场景 |
| **ib_atomic_bw** | RDMA Atomic | 测试原子操作带宽 | 特殊原子操作场景 |

**推荐组合**：
1. **带宽测试**：ib_write_bw、ib_read_bw、ib_send_bw（覆盖三种主要操作）
2. **延迟测试**：ib_write_lat、ib_read_lat（关键延迟指标）
3. **双向测试**：ib_write_bw -b（双向同时写）

### User's Answer
**需要全部perftest命令**：
- 带宽测试：ib_write_bw、ib_read_bw、ib_send_bw
- 延迟测试：ib_write_lat、ib_read_lat、ib_send_lat
- 双向测试：ib_write_bw -b 等

**设计要点**：
1. 测试框架支持所有perftest命令
2. 测试用例中指定具体使用哪个命令（如 `rdma_tool: "ib_write_bw"`）
3. 不同用例覆盖不同RDMA操作类型：
   - 压力测试：ib_write_bw（最高带宽）
   - 延迟测试：ib_write_lat/ib_read_lat
   - 长时间测试：ib_write_bw -b（双向稳定运行）

## Q10: 是否需要支持驱动切换测试？

### My Recommendation
驱动切换测试是指：在一次测试过程中切换驱动类型（如 kernel → dpdk → rdma），验证网卡在不同驱动下的稳定性。

| 场景 | 描述 | 测试目的 |
|------|------|----------|
| **单驱动测试** | 全程使用一种驱动（kernel/dpdk/rdma） | 验证特定驱动下网卡稳定性 |
| **驱动切换测试** | 测试过程中切换驱动（如先kernel测试，后切换到dpdk测试） | 验证驱动切换过程网卡是否正常、切换后稳定性 |

**推荐：单驱动测试为主，可选支持驱动切换测试**

设计：
1. 默认场景：单次测试全程使用一种驱动
2. 可选场景：提供驱动切换脚本，用于手动或特定用例执行驱动切换
3. 驱动切换测试作为独立用例类型（属于异常测试范畴）

### User's Answer
**支持多网卡 + 多驱动混合场景**：
- DUT可能有多块网卡或多端口网卡
- 每个网卡口可以独立配置驱动类型
- 组合示例：网卡0口(kernel驱动) + 网卡1口(dpdk驱动)
- 根据实际测试用例灵活组合

**设计要点**：
1. 配置支持多网卡定义：`nics: [{name: "eth0", driver: "kernel"}, {name: "eth1", driver: "dpdk"}]`
2. 每个网卡口独立配置：驱动类型、测试工具、测试参数
3. 用例可以指定测试哪些网卡口、使用哪种驱动组合
4. Tester端需要对应配置：多网卡口的流量发送端
5. 测试框架需要支持并行测试多个网卡口（或串行测试）

## Q12: 多网卡测试执行方式？

### Q13: 压力测试的核心测试目标是什么？

### My Recommendation
压力测试（Stress Test）的核心目标是在高负载条件下验证网卡稳定性：

| 测试目标 | 描述 | 验证指标 |
|----------|------|----------|
| **吞吐量极限** | 验证网卡能否达到额定带宽（或接近额定带宽） | 实测带宽 vs 额定带宽比例 |
| **丢包率验证** | 高负载下丢包率是否在可接受范围 | 丢包率 < 0.1%（可配置阈值） |
| **延迟稳定性** | 高负载下延迟是否稳定、抖动是否可控 | 延迟均值、延迟抖动范围 |
| **CPU负载** | 高负载下CPU使用率是否合理 | CPU使用率（不应100%饱和） |
| **长时间高压** | 能否持续高压运行不崩溃 | 持续运行时长、期间无错误 |

**推荐组合目标**：
1. 验证网卡在80%~90%额定带宽下稳定运行
2. 丢包率控制在可接受范围（如<0.1%或<1%）
3. 延迟抖动可控（如抖动范围<10%均值）
4. CPU使用率合理（如kernel模式<80%，dpdk模式可能更高）
5. 持续高压运行一定时长（如1小时、4小时、8小时）

### User's Answer
**压力测试核心目标**：

## Q18: 压力测试TCP还是UDP模式？

### My Recommendation
TCP和UDP是两种不同的传输协议，测试场景和关注指标不同：

| 协议 | 特点 | 测试关注点 | 适用场景 | 工具参数 |
|------|------|------------|----------|----------|
| **TCP** | 可靠传输、有重传机制 | 吞吐量、延迟、CPU负载、带宽利用率 | 标准业务场景、存储网络、数据库 | iperf3默认TCP模式 |
| **UDP** | 无连接、无重传、可丢包 | 丢包率、PPS、延迟抖动、吞吐量 | 视频流、实时通信、高频交易 | iperf3 -u |

**关键差异**：
- TCP模式：丢包会触发重传，关注吞吐量稳定性和重传率
- UDP模式：丢包不重传，关注丢包率和延迟抖动

**推荐：TCP和UDP都需要测试，用例分别覆盖**

用例设计：
```
压力测试（按协议）：
├── TCP压力测试系列：
│   ├── TCP吞吐量压力测试（带宽验证）
│   ├── TCP多流压力测试（多连接并发）
│   └── TCP长连接压力测试（连接稳定性）
│
└── UDP压力测试系列：
```

## Q20: 压力测试是否需要测试不同TCP窗口大小？

**不需要TCP窗口大小测试**：
- 主要关注网卡硬件稳定性，不是性能调优测试

**范围边界**：

## Q28: 长时间测试中断恢复机制是什么？

### My Recommendation
长时间测试（7x24小时甚至更长）可能遇到各种中断情况，需要设计恢复机制：

**可能的中断场景**：
| 中断类型 | 原因 | 影响范围 | 恢复难度 |
|----------|------|----------|----------|
| **流量工具中断** | iperf3/pktgen进程崩溃 | 流量停止，监控继续 | 低（重启工具） |
| **网络临时中断** | 网线松动、交换机重启 | 短时间通信中断 | 中（等待恢复） |
| **设备重启** | DUT或Tester系统重启 | 所有测试停止 | 高（需要重新初始化） |
| **Monitor中断** | Prometheus服务中断 | 监控数据丢失 | 低（恢复监控） |
| **人为干预中断** | 手动停止、紧急暂停 | 所有测试停止 | 需人工处理 |

**中断恢复策略选项**：

| 策略 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **自动重试恢复** | 检测中断后自动重新启动测试 | 无需人工干预，自动化 | 可能掩盖真正问题 | 自动化测试场景 |
| **记录中断+继续** | 记录中断点，恢复后继续运行 | 保留中断信息，继续测试 | 需要人工确认恢复 | 标准长时间测试（推荐） |
| **标记失败+终止** | 中断即标记测试失败，终止测试 | 严格判定，问题立即暴露 | 可能因小问题终止整个测试 | 高可靠性要求场景 |
| **忽略中断+继续** | 中断后忽略，继续测试 | 灵活，容忍临时中断 | 可能遗漏问题 | 宽松验证场景 |

**推荐：记录中断+继续**

设计要点：
1. 检测中断：监控流量工具进程状态、网络连通性
2. 记录中断：记录中断时间、中断类型、中断时长
3. 自动恢复：尝试自动恢复流量工具（如重启iperf3）
4. 继续测试：恢复后继续运行，总时长累计
5. 报告标注：测试报告中标注中断事件，便于分析

```yaml
# 中断恢复配置
interrupt_handling:
  strategy: "record_and_continue"  # 记录中断+继续
  auto_recovery: true              # 自动尝试恢复
  max_recovery_attempts: 3         # 最大恢复尝试次数
  recovery_interval: "5m"          # 恢复间隔（等待网络恢复）
  report_interrupts: true          # 报告中标注中断事件
```

**中断监控与恢复流程**：
```
检测中断 → 记录中断事件 → 尝试自动恢复 → 成功恢复继续 / 失败记录并通知
```

## Q32: 异常测试恢复验证标准是什么？

异常测试的核心是验证"异常发生后的恢复能力"，需要明确恢复验证标准：

**恢复验证指标**：

| 验证项 | 验证内容 | 验证方式 | 验证标准 |
|--------|----------|----------|----------|
| **链路状态恢复** | 链路是否恢复Up状态 | `ip link show eth0` | 状态为UP |
| **流量恢复** | 流量是否能正常传输 | iperf3测试带宽 | 带宽恢复到正常水平 |
| **错误计数** | 异常期间是否有异常错误计数 | `/sys/class/net/statistics/` | error计数恢复正常增长 |
| **内存状态** | 异常后内存是否恢复稳定 | Prometheus内存监控 | 内存无异常增长 |
| **进程状态** | 相关进程是否正常运行 | 进程检查 | 进程存活、无异常重启 |
| **连接恢复** | TCP连接是否正常建立 | 连接测试 | 新连接可正常建立 |

**恢复验证流程**：
```
触发异常 → 等待恢复 → 验证恢复状态 → 验证流量恢复 → 验证指标正常 → 测试通过
```

**推荐验证标准**：
1. 链路状态：恢复后状态为UP
2. 流量恢复：恢复后能正常传输流量（带宽达到目标）
3. 错误计数：异常期间的error计数需记录，但恢复后不再异常增长
4. 内存状态：异常后内存恢复稳定，无泄露
5. 连接恢复：恢复后能正常建立新连接

**恢复时间标准**：
- 链路恢复：down后up，预期10秒内恢复
- 流量恢复：恢复后30秒内流量恢复正常
- 驱动重载：驱动重载后30秒内网卡可用

```yaml
# 异常测试恢复验证配置
recovery_verification:
  link_status: "UP"               # 链路状态必须为UP
  traffic_recovery_time: "30s"    # 流量恢复时间
  bandwidth_after_recovery: "75%" # 恢复后带宽目标
  error_count_increase: "record"  # 记录异常期间error增长
  memory_stable: true             # 内存需稳定
  connection_test: true           # 连接测试需通过
```

## Q39: 业务流测试的驱动类型覆盖范围？

### My Recommendation
业务流测试需要覆盖哪些驱动类型？

| 驺动类型 | 业务流测试必要性 | 原因 | 必选/可选 | 推荐度 |
|----------|--------------------|------|-----------|--------|
| **kernel** | **必须覆盖** | kernel驱动是业务流最常见场景，Web/MySQL/NFS都是基于内核协议栈 | 必选 | ★★★★★ |
| **dpdk** | 可选覆盖 | DPDK场景业务流较少见（需特殊应用），可选 | 可选（P2） | ★★★★ |
| **rdma** | 可选覆盖 | RDMA业务流场景特殊（如RDMA存储），可选 | 可选（P2） | ★★★★ |

**推荐：kernel必选，dpdk和rdma可选**

原因：
1. **kernel驱动**：Web服务、MySQL数据库、NFS存储都是基于内核TCP/IP协议栈，是最常见业务场景，必须覆盖
2. **dpdk驱动**：DPDK场景业务流需要特殊应用（如DPDK-nginx、DPDK-MySQL），实现复杂，可选覆盖
3. **rdma驱动**：RDMA业务流场景特殊（如RDMA存储iSER、NVMe-oF），可选覆盖

**与之前测试类型的必选/可选策略略有不同**：
- 压力测试：kernel/dpdk/rdma都覆盖（根据用例选择）
- 长时间测试：kernel/dpdk/rdma都覆盖
- 异常测试：kernel/dpdk必选，rdma可选
- 混合数据测试：kernel/dpdk必选，rdma可选
- **业务流测试：kernel必选，dpdk/rdma可选**（业务流主要基于内核协议栈）

**设计要点**：
```yaml
# 业务流测试驱动类型配置

  - dpdk: "需要特殊DPDK应用支持（如DPDK-nginx）"
  - rdma: "需要RDMA业务应用支持（如NVMe-oF存储）"
```
